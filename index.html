Chapter 01

Foundations

~30 minutes

1.1 — What Claude Code Actually Is

Claude Code is Anthropic's official CLI (command-line interface) that lets Claude operate directly in your terminal and IDE. It can read files, write files, run commands, search the web, and build entire projects.

Paradigm Shift

Instead of copying code from a chatbot into your files, Claude Code works directly in your project — reading, writing, and executing like a developer sitting at your machine.

1.2 — Installation

What you need:

  • A Claude Pro plan minimum ($20/month)
  • A computer with a terminal (Mac, Windows, or Linux)
  • An IDE: VS Code or Cursor (the workspace where you'll operate)

How to install:

  • One terminal command: npm install -g @anthropic-ai/claude-code
  • Node.js must be installed first — it's something you install on the laptop so the AI can run JavaScript-based projects. Download from nodejs.org

How to authenticate:

  1. Run claude in your terminal
  2. It opens a browser window for you to log in with your Anthropic account
  3. Once authenticated, you're ready

1.3 — IDE Setup

VS Code or Cursor is the workspace where you'll operate. The terminal inside the IDE is where Claude Code runs.

Terminal mode (running claude directly in a standalone terminal) unlocks more features, but IDE terminal is fine for most work.

1.4 — The 4 Permission Modes

Cycle through them with Shift+Tab.

ModeWhat It DoesWhen to Use
Ask Before EditsClaude asks permission before every file changeClient work where you need to review everything
Edit AutomaticallyClaude edits files without asking, but still asks for commandsYour own projects — balanced speed/safety
Plan ModeClaude researches and plans but doesn't executeComplex builds, research phase, enterprise setups
Bypass PermissionsClaude does everything without askingBrand new throwaway projects ONLY — never on client work

1.5 — Context Management

Tokens are your "budget" per conversation. Think of them as the AI's working memory.

  • /context — Check how much of your budget is used
  • /compact — Compress the conversation to free up space (Claude summarizes everything so far)
  • /clear — Wipe the conversation completely and start fresh
  • /cost — See how much money this session has cost
Why Context Matters

As context fills up, Claude's output quality degrades. It starts forgetting instructions, making mistakes, and losing focus. Managing context is an operational skill.

1.6 — The Fundamental Workflow Loop

Task → Execute → Verify → Iterate

This is the #1 thing to internalize. Most people fail because they skip verification. They ask Claude to build something, assume it worked, and move on. Then they discover 10 problems stacked on top of each other.

  1. Task — Tell Claude exactly what you want
  2. Execute — Let Claude build it
  3. Verify — Check the output yourself (open in browser, run the code, test it)
  4. Iterate — Tell Claude specifically what to fix

Chapter 02

The CLAUDE.md Mastery Guide

~45 minutes

This is the core of the entire manual. CLAUDE.md is the single most important file for controlling Claude Code's behavior.

2.1 — What CLAUDE.md Actually Does

The "ship metaphor": CLAUDE.md is injected at the very start of every session, before anything you type. A tiny inaccuracy at launch = massive deviation at destination. It steers the AI.

From Boris Cherny (Creator of Claude Code)

"Anytime we see Claude do something incorrectly we add it to the CLAUDE.md, so Claude knows not to do it next time." It's a living document.

Token efficiency: Without CLAUDE.md, Claude has to read your entire project every time to understand what it's working with. With CLAUDE.md, it gets a summary upfront, saving massive tokens and increasing quality.

2.2 — The Six-Tier Memory Hierarchy

The tutorial covers 3 layers. The official docs reveal 6. But before they make sense, you need to understand Two foundational concepts.

Foundation: Hidden Folders on macOS

On a Mac, any file or folder whose name starts with a dot (.) is hidden. You can't see it in Finder unless you press Cmd + Shift + . to reveal hidden files.

  • .claude/ — hidden
  • .DS_Store — hidden
  • CLAUDE.md — visible (no dot at the start)

Foundation: The Tilde (~) vs the Dot-Slash (./)

These two symbols tell you WHERE on your Mac a file lives:

~   = your HOME folder = /Users/your-username/
./  = whatever project folder you're currently in

So:

~/.claude/CLAUDE.md     →  /Users/your-username/.claude/CLAUDE.md
                           This is in your HOME. It has nothing to do
                           with any project. It's YOUR personal config.

./CLAUDE.md             →  /Users/your-username/Desktop/my-project/CLAUDE.md
                           This is inside your project folder.
                           It belongs to THIS project only.
Common Confusion

The tilde is NOT on your Desktop. Your home folder is one level above Desktop.

/Users/your-username/ ← this is ~ (HOME) ├── Desktop/ │ └── my-project/ ← this is ./ (your project) ├── Documents/ ├── Downloads/ └── .claude/ ← this is ~/.claude/ (GLOBAL, hidden)

Two .claude Folders — Different Places, Different Jobs

This is the most confusing part. There are TWO .claude folders with the same name but in completely different locations:

/Users/your-username/ ← HOME (~) │ ├── .claude/ ← GLOBAL .claude │ │ Created automatically by Claude Code. │ │ Lives in your HOME folder. │ │ Affects ALL your projects. │ ├── settings.json (your global settings) │ ├── projects/ (auto-memory per project) │ └── CLAUDE.md (your personal preferences — create if needed) │ └── Desktop/my-project/ ← YOUR PROJECT (./) │ ├── .claude/ ← PROJECT .claude │ │ Created by you or Claude Code. │ │ Lives INSIDE the project. │ │ Affects ONLY this project. │ ├── settings.json (project-specific settings, e.g. MCP servers) │ └── rules/ (split rules when CLAUDE.md gets too long) │ └── *.md │ ├── CLAUDE.md ← the project brain (visible, shared via git) └── CLAUDE.local.md ← your personal overrides (visible, NOT shared)

The 6 Tiers — From Strongest to Weakest

TierNameWhere It LivesWho Sees ItWhat It's For
1Managed PolicySystem-level path (pushed by IT)Every employee"Company law" — nobody can override it. Enterprise only.
2Project Memory./CLAUDE.md (project folder, visible)Whole team via gitThe main project brain. Most important for daily work.
3Project Rules./.claude/rules/*.md (project, hidden)Whole team via gitSame as Tier 2, split into files when CLAUDE.md gets too long.
4User Memory~/.claude/CLAUDE.md (home, hidden)Only youYour personal preferences across ALL projects.
5Project Local./CLAUDE.local.md (project, visible)Only you (gitignored)Your personal overrides for ONE specific project.
6Auto Memory~/.claude/projects/<id>/memory/ (home, hidden)Only ClaudeNotes Claude writes to itself. You never touch this.

What Happens When Rules Conflict?

Higher tier wins. Always.

Example conflict:

  Tier 4 (your personal ~/.claude/CLAUDE.md):     "Use tabs"
  Tier 2 (project ./CLAUDE.md):                   "Use spaces"

  → Claude uses SPACES, because Tier 2 beats Tier 4.
    The project's rules override your personal preferences.

Another example:

  Tier 2 (project ./CLAUDE.md):                   "Use console.log for debugging"
  Tier 1 (company managed policy):                 "Never use console.log"

  → Claude follows the COMPANY rule. Tier 1 beats everything.
Key Insight Not in Most Tutorials

More specific instructions beat broader ones. Project rules override your personal CLAUDE.md.

2.3 — The Golden Rules for Writing CLAUDE.md

  • Keep it short: Under 60-80 lines for project CLAUDE.md. Under 15 lines for global. Claude's system prompt already has ~50 instructions. The model can reliably follow ~150-200 total. Every line you add competes with the actual work.
  • Most important rules at the TOP: Primacy bias — AI remembers beginnings and endings best, middles worst.
  • Bullet points, short headings: High information density, not paragraphs.
  • Only include what Claude can't figure out by reading the code: Don't describe what's obvious. DO include: build commands Claude can't guess, non-standard conventions, gotchas, architecture decisions.
  • Don't dump API docs or style guides: Causes 10,000+ token bloat, reduces quality. Instead, tell Claude where to find docs when needed.
  • Don't write vague aspirational rules: "Be smart" or "Make no mistakes" wastes tokens. Be specific: "Always use ES modules, never CommonJS."
  • If Claude makes the same mistake 2-3 times, add a rule: The #1 way to improve your CLAUDE.md over time.
  • Never write negative-only constraints: "Never use --foo flag" causes paralysis. Always provide alternatives: "Use --bar instead of --foo."
  • Periodically review and prune: CLAUDE.md grows over time with unnecessary additions. Treat it like living code.
  • Use emphasis for critical rules: "IMPORTANT" or "YOU MUST" for rules that absolutely cannot be broken.

2.4 — What Goes INSIDE a CLAUDE.md (the WHY-WHAT-HOW Framework)

SectionPositionContents
WHATTop (most important)Project name, one-line purpose, tech stack (just names), project structure map
HOWMiddleBuild/run commands (the ones Claude can't guess), testing commands, deployment instructions
WHYClarifying intentKey architectural decisions and WHY they were made, gotchas, non-obvious behaviors
DON'TSBottom (guardrails)Files Claude must never modify, patterns to avoid, specific past mistakes

2.5 — The /init Command and When to Use It

  • Reads entire codebase and auto-generates a CLAUDE.md summary
  • Great starting point for any new project folder
  • BUT: Always manually review and prune
  • Best approach: Run /init, then spend 2-3 minutes editing the output to match the golden rules

2.6 — The @import System

CLAUDE.md files can import other files, keeping your CLAUDE.md lean while giving Claude access to detailed docs on demand:

See @README.md for project overview
Git workflow: @docs/git-instructions.md

Max 5 levels deep.

3.3 — Rules Folder Details

  • All .md files in .claude/rules/ auto-load into context
  • Can use YAML frontmatter with paths: to scope rules to specific files

Chapter 04

Core Workflows

~45 minutes

4.1 — Website/Landing Page Design (The Screenshot Loop)

The most effective method for building visual designs with Claude Code:

  1. Find design inspiration (godly.website, Dribbble, or any site you like)
  2. Take a full-page screenshot (browser DevTools → Cmd+Shift+P → "Capture full size screenshot")
  3. Resize image to under 5MB
  4. Give screenshot + your brand info to Claude: "Rebuild this design with my content"
  5. Claude builds it. Take a screenshot of the result.
  6. Compare side by side. Tell Claude what to fix via voice (3x faster than typing)
  7. Repeat until 95%+ accurate (usually 3-5 rounds, ~10 minutes total)
Pro Tip: Voice Transcription

Use FN key (Mac) or WhisprFlow for dictation. You speak 3x faster than you type. Boris and multiple power users emphasize this as a game-changer.

4.2 — Full-Stack App Building Workflow

  1. Voice your requirements → Claude captures everything
  2. Switch to Plan Mode (Shift+Tab twice) → Claude researches and creates a structured plan
  3. Review the plan → Go back and forth until you're happy with architecture decisions
  4. Switch to Edit mode → Claude builds while you set up external services
  5. Test in browser → Verify each feature works
  6. Provide specific feedback → "Make the header smaller, change font to serif, add loading state"
  7. Deploy → Netlify/Vercel push
Key Insight

"1 minute of planning saves 10 minutes of building." Without plan mode, a complex app took 35+ minutes. With plan mode, 10-20 minutes.

4.3 — Automation/Workflow Building

  1. Define the workflow in plain language (voice or text)
  2. Create the skill structure: .claude/skills/skill-name/SKILL.md + scripts/
  3. SKILL.md is the orchestrator: front matter tells Claude when to use it, body is the step-by-step checklist
  4. Test with small dataset first (25-100 items)
  5. Add timing instrumentation ("tell me how long each step took")
  6. Scale up once quality is confirmed
  7. Optionally deploy as API via Modal ($5 free credits)
Security Warning

Never charge money for an AI-built app without a security audit. Keep URLs non-obvious. For internal/team tools this is less critical.

Chapter 05

Skills, MCP & Sub-Agents

~30 minutes

5.1 — Skills (The Most Economically Valuable Feature)

Automated workflows that Claude can execute. Think of them as "recipes" — you write the steps once, Claude follows them every time.

Business Value

A skill that saves 30 minutes of manual work, running 10 times a day, saves 5 hours daily. 1,000 leads were scraped in 87 seconds vs 30+ minutes manually.

SKILL.md Structure:

---
name: skill-name
description: What this does and WHEN to use it (include trigger words)
allowed-tools: Bash, Read, Write, WebFetch
---
# Steps
1. First do X
2. Then do Y
3. Verify by Z

Key Details:

  • Only the frontmatter (~60 tokens) loads into context at startup
  • Full instructions load ONLY when the skill is invoked
  • This is 100x more efficient than putting everything in CLAUDE.md
  • Skills can use $ARGUMENTS for dynamic inputs

5.2 — MCP Servers (Other People's Skills)

Pre-built integrations that give Claude access to external tools (browsers, email, databases, task managers).

Token Cost Reality

Each MCP tool costs 1,000-1,600 tokens just being loaded. A full MCP like ClickUp can consume 20,000 tokens before you even start working. That's 10% of your context gone.

The Decision Rule:

  • Use MCP for quick prototyping (5-minute setup, test if the approach works)
  • If you use it more than a few times → convert to a skill
  • Always-worth-it MCPs: Chrome DevTools (speed), Supabase (database)
  • Check token impact with /context after installing any MCP

5.3 — Sub-Agents (Parallel Workers)

Separate Claude instances that the main Claude spawns to handle specific tasks. Each has its own isolated context window.

The Three Standard Sub-Agents:

AgentPurposeKey Benefit
ResearchHeavy research, web scraping, data gatheringReturns only summary — saves 50x on tokens
ReviewerZERO project context, reviews code objectivelyUnbiased critique without knowing your intent
QARuns tests, checks if things workReports pass/fail without polluting parent context
Critical Math

10 sub-agents at 95% success rate = only 59% chance ALL succeed (probabilities multiply). Test at small scale first (100 items) before scaling to 1,000+.

Chapter 06

Advanced Patterns

~20 minutes

6.1 — Parallelization with Sub-Agents

Serial: Process 100 emails one by one → 36 seconds

Parallel: 10 sub-agents process 10 emails each → same work done simultaneously

Real example: 1,000 emails classified in ~60 seconds with 10 parallel sub-agents.

  • When to parallelize: When you have many independent items to process (emails, leads, pages, files)
  • When NOT to: Simple single tasks where spawning sub-agents takes longer than just doing it

6.2 — Agent Teams (Experimental, High Cost)

Multiple Claude instances that can communicate DIRECTLY with each other (not just through a parent). They share a task list and coordinate autonomously.

Enable: Add "env": {"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"} in settings.json.

When to use:

  • Design exploration: Spawn 3 design agents, each creates a fundamentally different approach
  • Research synthesis: Spawn 3 research agents, each investigates a different angle
  • Security audit: Spawn 10 scanner agents across a large codebase
Cost Warning

Each agent is a separate Claude instance. 7x more expensive than single-thread. "Trade money for time EXPLICITLY." Real example: a security audit cost $80, used 1.3 million tokens, found 15 issues in minutes vs weeks for a human team.

6.3 — Git Worktrees (Parallel Feature Development)

Multiple isolated working directories that share the same git history. Each directory can have its own Claude instance working on a different feature.

Why Worktrees Matter

Without worktrees, two Claude instances editing the same file will conflict. With worktrees, each works in its own folder on its own branch. Real example: Built 4 pages simultaneously, then merged them together.

Rule: Specify in your prompts/CLAUDE.md: "Work only in your worktree folder. Don't modify files outside it." This prevents merge conflicts.

Chapter 07

Enterprise & Business Setup

~30 minutes

7.1 — How to Set Up Claude Code for a Company

Step 1: Choose the Right Plan

  • Team plan: For small teams, shared billing, basic admin controls
  • Enterprise plan: SSO, managed settings, compliance API, IT controls
  • Premium seats = Claude Code access. Standard seats = no Claude Code.

Step 2: Deploy Managed Settings (Enterprise Only)

The IT/DevOps team deploys managed-settings.json at the system level:

  • macOS: /Library/Application Support/ClaudeCode/managed-settings.json
  • Linux: /etc/claude-code/managed-settings.json
  • Windows: C:\Program Files\ClaudeCode\managed-settings.json

This is the enterprise CLAUDE.md. It CANNOT be overridden by any user or project settings.

Steps 3-5: Project Configuration

  1. Create the project CLAUDE.md with WHY-WHAT-HOW framework
  2. Set up shared .claude/ configuration (settings.json, rules/, skills/, .mcp.json)
  3. Configure hooks for quality guarantees (deterministic, unlike CLAUDE.md rules)

7.2 — The Hooks System (Deterministic Automation)

Critical Difference

Unlike CLAUDE.md instructions (which Claude may sometimes ignore), hooks are DETERMINISTIC — they ALWAYS execute.

EventWhen It FiresCan Block Claude?
SessionStartSession beginsNo
UserPromptSubmitYou send a messageYes
PreToolUseBefore Claude uses a toolYes (can deny or modify)
PostToolUseAfter a tool succeedsNo (feedback only)
StopClaude finishes respondingYes
SubagentStartSub-agent spawnedNo (inject context)
TaskCompletedTask marked completeYes (quality gate)

7.3 — Business Scenario Playbooks

Scenario A: SaaS Startup (8 developers, skeptical CTO)

  1. Start with a 2-day paid audit ($5,000) — measure current workflow baselines
  2. Identify 3 highest-ROI use cases (usually: test writing, boilerplate, documentation)
  3. Create CLAUDE.md for their repo using WHY-WHAT-HOW framework
  4. Set up .claude/settings.json with conservative permissions
  5. Train 3 pilot developers for one week
  6. Measure: PRs per week, time-to-merge, bug rate
  7. Present results to CTO with before/after data → convert to full rollout

Scenario B: Marketing Agency (15-person team, no developers)

  1. Focus on skills: email classification, lead scoring, report generation
  2. Set up Team plan (not Enterprise)
  3. Create 3-5 skills for their most repetitive workflows
  4. Train team on invoking skills (they don't need to understand internals)
  5. Measure: hours saved per week per person × hourly cost = ROI
  6. Monthly retainer for optimization and new skill creation

Scenario C: Enterprise (200+ developers, compliance requirements)

  1. Work with IT to deploy managed-settings.json via MDM
  2. Set up SSO integration
  3. Create company-wide CLAUDE.md standards (managed policy tier)
  4. Deploy hooks: PostToolUse for linting, PreToolUse for security scanning
  5. Pilot with 20 developers across 3 teams for 4 weeks
  6. Measure with full telemetry (Prometheus + dashboards)
  7. Phased rollout over 12 weeks with weekly ROI reporting

7.4 — Security Best Practices for Client Work

  • Never run Claude Code as root
  • Block access to .env, ~/.ssh/, secrets/ directories
  • Deny curl/wget by default (prevents data exfiltration)
  • Enable sandbox mode for bash commands
  • For monetization: have a developer review authentication and authorization at minimum

Chapter 08

ROI, Pricing & Pitching AI Solutions

~30 minutes

8.1 — The Honest Data Landscape

The Positive Case:

  • Developers complete tasks 55% faster with AI assistance (GitHub/Microsoft study)
  • PR turnaround dropped 75% — from 9.6 days to 2.4 days (GitHub Copilot data)
  • 84% of developers now use AI tools regularly (Stack Overflow 2025)
  • Teams with heavy AI use complete 21% more tasks and merge 98% more PRs

The Cautionary Case (use this for credibility — never hide it):

  • Experienced open-source developers were 19% SLOWER with AI (METR randomized controlled trial)
  • 75% of orgs see NO measurable performance gains despite individual improvements
  • Only 5% of companies achieve substantial AI value at scale
  • 42% of companies abandoned most AI initiatives in H1 2025
What This Means

A realistic, defensible productivity improvement range is 10-30%, not the 55% headline number. The tool alone is NOT the value — the implementation, training, and workflow redesign is where real ROI lives. This is exactly why consultants have a massive market opportunity.

8.2 — Core ROI Formulas (The 3 You Actually Need)

Formula 1: Basic Time-Savings ROI
Monthly Value Created = (Hours Saved per Week × 4.33) × Hourly Rate
Monthly Net Benefit = Monthly Value Created - Monthly Tool Cost
Annual ROI = (Annual Net Benefit / Annual Tool Cost) × 100%
Worked example:
Developer salary: $120K/yr → loaded cost $162K → hourly rate $78
Hours saved: 3/week (conservative)
Monthly value: 13 hrs × $78 = $1,012
Monthly tool cost: $20 (Pro plan)
Monthly net benefit: $992
Annual ROI: 4,962% | Payback: < 1 week
Formula 2: FTE Equivalence (Best for Executive Pitches)
FTE Equivalence = Total AI-Automated Hours per Month / 173.33
Worked example:
10 developers, each saving 3 hrs/week = 130 hrs/month saved
FTE Equivalence: 0.75 FTE
Translation: "Claude Code gives your 10-person team the output of 10.75 developers for the cost of 0.07 of a developer."
Formula 3: Cost-Per-Task Before/After (Most Persuasive)
Savings Per Task = (Time Before × Hourly Rate) - ((Time After × Hourly Rate) + AI Cost)
Annual Savings = Savings Per Task × Tasks Per Year
Worked example:
Task: Writing unit tests
Before: 4 hrs × $78 = $312
After: 1.5 hrs × $78 + $0.50 = $117.50
Savings per occurrence: $194.50 (62% reduction)
If done 3x/week: $30,342 annual savings per developer

9.1 — Common Failures and How to Fix Them

ProblemSymptomFix
Token limit hit Claude stops or gives degraded output Run /compact. If critical, /clear and start fresh with focused prompt
Context pollution Claude confuses different parts or repeats mistakes New session (/clear). Copy only essential context to new prompt
MCP won't connect Error on startup or "tool not available" Check .mcp.json syntax. Restart Claude Code. Verify MCP server is running
CLAUDE.md ignored Claude does what you told it not to Move critical rules to TOP (primacy bias). Use "IMPORTANT:" prefix. Convert to hook if persistent
Sub-agent garbage Sub-agent output is wrong or incomplete Simplify the task. Give ONE clear job. Check that agent spec has right tools listed
Session hangs No output for 30+ seconds Wait 30s. If stuck, Ctrl+C and retry. Check /context — may be near token limit
Wrong mode Claude asks permission for everything (or never asks) Check status bar. Press Shift+Tab to cycle modes
Hook blocks everything Claude can't complete any action Check settings.json hooks section. Test hooks with simple commands first

9.2 — Cost Management as an Operational Discipline

Cost Optimization Strategies:

  1. Use /compact proactively — don't wait until you hit limits. Compact after each major task
  2. Start new sessions for new tasks — don't carry context from one task into another
  3. Use Sonnet for sub-agents — cheaper and fast enough. Reserve Opus for main session
  4. Convert frequently-used MCPs to skills — 60 tokens vs. 1,600 tokens per tool
  5. Remove unused MCPs — each loaded tool eats context even if unused
  6. Keep CLAUDE.md under 60 lines — every line costs tokens on EVERY prompt

Monthly Cost Estimation:

Warning Signs ROI Isn't Being Captured:
